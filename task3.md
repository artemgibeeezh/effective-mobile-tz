# Задание 3. Архитектура PUSH-уведомлений

## Блок-схема (словами)

1. **Мобильное приложение** регистрируется в FCM (Firebase Cloud Messaging) или APNS (Apple Push Notification Service) и получает push-токен.

2. Приложение отправляет полученный токен на бэкенд (микросервис уведомлений) вместе с идентификатором пользователя.

3. Микросервис уведомлений сохраняет токен в базе данных.

4. Другие микросервисы (корзина, заказы, реклама) при наступлении события (просрочка корзины, отмена заказа, рекламная рассылка) отправляют событие в брокер сообщений (например, Kafka или RabbitMQ).

5. Микросервис уведомлений читает события из брокера, определяет нужного пользователя, формирует текст и заголовок уведомления.

6. Микросервис уведомлений отправляет запрос в FCM/APNS.

7. FCM/APNS доставляет уведомление на устройство пользователя.

8. При клике на уведомление приложение открывает соответствующий экран (корзина, заказ, акция).

## Компоненты

- Мобильное приложение (iOS / Android)
- Firebase Cloud Messaging / Apple Push Notification Service (шлюзы отправки)
- Микросервис уведомлений
- Брокер сообщений (Kafka / RabbitMQ)
- База данных (токены устройств, история отправок)
- Админ-панель для ручных рассылок
